## 智能公文「起草 + 格式化」问题分析与优化方案

### 一、🔴 严重 Bug（需立即修复）

#### 1. RAF 缓冲区未正确重置（前端 SmartDocView.tsx）

- **位置**: SmartDocView.tsx ~L1095-1161
- **问题**: `_streamBufRef` 在 `resetStreamingText()` 中没有清空。如果 SSE 重连或切换文档，旧缓冲区的残留文本会与新流拼接导致内容重复。
- **修复**: 在 `resetStreamingText()` 中加入 `_streamBufRef.current = ""`

#### 2. `isAiThinking` 状态未在流结束时保证重置

- **位置**: SmartDocView.tsx ~L1207-1208, 2593-2615
- **问题**: reasoning 事件设 `isAiThinking = true`，但 `onDone` 回调里没有保证 `setIsAiThinking(false)`。导致 AI 思考面板可能永远显示。
- **修复**: 在 `onDone` 里明确 `setIsAiThinking(false)`

#### 3. 多轮续写 Token 上限硬编码（后端 documents.py）

- **位置**: documents.py ~L3488
- **问题**: `_completion_tokens >= 15500` 硬编码为 qwen3-32b 的 16384 max。若切换模型（如 GPT-4/DeepSeek），此阈值完全失效，导致不必要的续写或漏掉续写。
- **修复**: 从 Dify 服务配置中读取 `max_tokens`，或作为环境变量配置

#### 4. 路径遍历安全风险（后端 documents.py）

- **位置**: documents.py ~L3007
- **问题**: `doc.source_file_path` 直接用 `Path().read_bytes()` 读取文件，未校验路径是否在 `UPLOAD_DIR` 安全目录内。存在路径遍历攻击风险。
- **修复**: 验证 `source_path.resolve()` 在 `settings.UPLOAD_DIR / "documents"` 范围内

#### 5. Dify 格式化 JSON 括号解析器不健壮（后端 client.py）

- **位置**: client.py ~L1201-1240
- **问题**: `_try_parse_incremental_suggestions()` 用简单括号计数 `{` `}`，遇到字符串中包含 `}` 如 `{"reason": "text with } bracket"}` 会错误截断。
- **修复**: 增加字符串状态机追踪（处理 `"..."` 内的转义括号）

---

### 二、🟡 高优先级问题（影响核心体验）

#### 6. 知识库检索阻塞 SSE 流（后端 documents.py）

- **位置**: documents.py ~L3073-3200
- **问题**: 起草阶段检索知识库时，同步 `await` 完成后才开始发送 SSE 事件。每个知识库集合 1-3 秒，用户在此期间看到空白等待。
- **优化**:
  - 立即发送 `progress` 事件（"正在检索知识库..."）
  - 用 `asyncio.create_task()` 并行检索 + Dify 流开始
  - 或至少给个进度百分比

#### 7. 前端 SSE 空闲超时仅 5 分钟（前端 documents.ts）

- **位置**: documents.ts ~L281
- **问题**: `SSE_IDLE_TIMEOUT = 5 * 60 * 1000`，对于 50 页复杂文档的起草/格式化（含 reasoning 可能 5-10 分钟），超时会中断作业。
- **修复**: 增加到 15 分钟，或根据 stage 类型动态调整（draft/format 给更大超时）

#### 8. `aiStructuredParagraphs` 与 `acceptedParagraphs` 竞态

- **位置**: SmartDocView.tsx ~L1071, 1074, 1418-1420
- **问题**: 两个状态同时存在且优先级不明确。`handleAcceptAll()` 用 `setTimeout(0)` 清理 `acceptedParagraphs` 是个 hack，在 React 18+ 批量更新下可能产生一帧的闪烁。
- **优化**: 用单一状态 + `status` 字段区分（如 `paragraphs: {data: [], source: "ai"|"accepted"}`）

#### 9. format_suggest 不返回修改后的段落

- **位置**: documents.py ~L4194-4232
- **问题**: 与 draft/format 阶段（返回完整段落+变更标记不同），format_suggest 仅返回文字建议，用户必须手动对照修改。体验割裂。
- **优化**: 除返回建议外，同时返回 `modified_paragraphs` 带 `_change` 标记，允许"一键应用"

#### 10. 规则引擎低置信度段落静默使用默认值

- **位置**: documents.py ~L3815-3823
- **问题**: 规则引擎覆盖率 ≥80% 时跳过 LLM，低置信度段落直接用模板默认值填充。用户不知道哪些段落是"猜的"。
- **优化**: 在结果中标记这些段落（如 `_confidence: "low"`），前端用浅黄色背景提示用户手动确认

---

### 三、🟠 中等优先级（体验和稳定性）

#### 11. Markdown 解析丢失多行段落内聚性

- **位置**: documents.py ~L738-787
- **问题**: `text.split('\n')` 逐行处理，不做段落合并。连续正文行被拆成多个独立段落，导致格式化后行距异常。
- **优化**: 实现段落边界检测（空行分割），连续 body 行合并为一个段落

#### 12. processingLog 无限增长

- **位置**: SmartDocView.tsx ~L1162-1165
- **问题**: 处理日志数组从不裁剪，长时间编辑会积累数千条，影响内存。
- **修复**: 限制上限 100 条，FIFO 淘汰旧记录

#### 13. 撤销/重做在 RAF 缓冲区下不安全

- **位置**: SmartDocView.tsx ~L1480-1497
- **问题**: 触发 undo 时 `_pendingParasRef` 中可能仍有待 flush 的段落。RAF 在 undo 后触发，会把旧数据追加到恢复后的状态。
- **修复**: `applySnapshot()` 中增加清空 `_pendingParasRef.current = []` 和取消 `_paraRafRef`

#### 14. 预设分类切换后 selectedPresetId 可能悬空

- **位置**: SmartDocView.tsx ~L1217, 3806-3850
- **问题**: 切换预设分类 filter 后，之前选中的预设可能不在当前分类下，但仍被保留使用。
- **修复**: 切换 filter 时重置 `setSelectedPresetId(null)` 或自动选中第一个

#### 15. AbortError 被静默吞掉

- **位置**: documents.ts ~L223
- **问题**: 用户点击取消时，AbortError 不触发 `onError` 回调，前端不知道处理已中止。
- **修复**: 区分处理——`onError("用户已取消")` 或特殊回调

#### 16. StructuredDocRenderer 无虚拟化

- **位置**: StructuredDocRenderer.tsx
- **问题**: 段落 >1000 条时直接 DOM 渲染，可能导致严重卡顿。
- **优化**: 引入 `react-virtualized` 或 `react-window` 做滚动虚拟化

#### 17. 自定义预设存 JSON 文件无数据库持久化

- **位置**: docformat/custom_presets.json
- **问题**: 用户创建的格式预设保存在 JSON 文件中，多实例部署时不同步，容器重启可能丢失。
- **优化**: 迁移到 PostgreSQL 表（`format_presets`），与用户关联

---

### 四、🔵 架构级优化建议

#### 18. 起草阶段增加"大纲生成→确认→正文展开"两步流程

- **现状**: 用户输入提示词一次性生成全文，缺乏中间可控环节。
- **优化**: 第一步 LLM 生成大纲（标题+章节结构），用户确认/编辑后，第二步展开正文。减少重新生成次数，提高命中率。
- **实现**: 新增 `outline` SSE 事件类型，前端增加大纲确认 UI

#### 19. 格式化引入"规则引擎 + LLM 混合"可视化

- **现状**: 规则引擎和 LLM 两条路径在后端静默选择，用户完全无感知。
- **优化**:
  - 前端显示 "规则引擎处理了 X 段 / LLM 处理了 Y 段"
  - 低置信度段落标黄色，UI 提供"让 AI 复检"按钮
  - 用户能看到每段的 `confidence` 和 `style_source`

#### 20. 统一段落数据流架构

- **现状**: `aiStructuredParagraphs`、`acceptedParagraphs`、`editHistory` 三套状态相互交叉，逻辑复杂。
- **优化**: 引入 `useReducer` 或状态机管理段落生命周期：
  ```
  idle → streaming → preview → accepted → editing → saved
  ```

#### 21. 导出格式一致性

- **现状**: 屏幕预览用 HTML/CSS 渲染，PDF 导出走 DOCX→PDF 转换，两者排版可能不一致。
- **优化**:
  - 导出前增加预览对比窗口
  - 或统一走 HTML→PDF（WeasyPrint）路径

#### 22. 增量格式化的 \_index 验证和告警

- **位置**: documents.py ~L4031-4040
- **问题**: AI 返回越界 `_index` 时静默丢弃，用户不知道哪些段落被跳过。
- **修复**: 记录跳过数量，SSE 发送 warning 事件，前端 toast 提示
